### Day 3 - 生成器 + yield
📅 完成日期:2026-04-19
📝 笔记:generator 本质是"可暂停可恢复的函数"——函数体里出现 yield 就变成 generator,调用它函数体一行不跑,只返回 generator 对象。每次 next 跑到下个 yield 就冻结,下次从 yield 后继续。for 循环其实是"自动 next + 自动捕获 StopIteration"。yield from 是把子 iterable 整个透传给上层调用方。generator 双向通信(send/throw)就是 asyncio 协程的前身。作业"流式读大文件"踩了 3 个典型坑:(1) read_lines 里只写 print 忘了 yield → 函数直接变普通函数;(2) yield line 先、rstrip 后 → rstrip 对已吐出的值无效(yield 是暂停键);(3) `line.rstrip("\n")` 单独一行 → str 不可变,返回值不接等于白跑。最终一行 `yield line.rstrip("\n")` 搞定。filter_contains 没用 yield from 而用了 for+if+yield,因为"过滤"场景 yield from 不合适(它是整批透传)。

### Day 4 - 迭代器协议
📅 完成日期:2026-04-19
📝 笔记:迭代器协议就是 Python 和容器之间的合同:实现 `__iter__` + `__next__` 就能被 for 遍历。`for x in obj` 等价于 `it = iter(obj); while True: try x = next(it) except StopIteration: break`。**Iterable vs Iterator 是今天最容易混的点**:list/tuple/str 是 Iterable(能 for 不能 next),iter(lst) 返回的才是 Iterator;Iterator 的 `__iter__` 必须返回 self,判别方法 `iter(x) is x` 为 True 就是 Iterator;Iterator 用一次就废(昨天 generator 只能遍历一次同病根)。generator 其实是迭代器协议的语法糖——自动满足协议,状态冻结/恢复由 Python 内建。作业踩的最大坑:**class Iterator 的 `__next__` 不能写 while(Python 会自动反复调),但 generator 的函数体必须 while True 包 yield(否则跑到底就 StopIteration)**——两种模式循环位置正好相反。对比结论:class 版 11 行 / generator 版 5 行,但加 `.reset()` class 秒赢 generator 做不到。

### Day 9 - async 深入
📅 完成日期:2026-04-21
📝 笔记:补 Day 8 遗留 `create_task` 开头 —— "起步发车,你想等就上车":和 `gather` 的区别是 gather 必须当场 await,create_task 可以先发后等,"fire-and-forget" 用于后台任务(心跳/LLM 流式)。**`await t1` 期间不是"只有 t1 在跑",事件循环里所有已调度的 task 都在跑**,完成顺序只看各自耗时(Q1 踩坑:以为 await 谁谁先完成,实际短的先)。**`Semaphore` 核心价值不是快是"活下来"**:甜点公式 `rate × 耗时 × 0.5`,四个典型场景用值记了个表(OpenAI 500rpm→10-20 / Prometheus→10-20 / 本地 1000 JSON→50-100 因为 FD + aiofiles 线程池上限 / MySQL 连接池 20→留 15 buffer)。`wait_for(coro, timeout=N)`:超时抛 `asyncio.TimeoutError`,内部原理就是 `create_task + cancel`。**异常类型踩坑**:`httpx.AsyncClient(timeout=3.0)` 抛 `httpx.TimeoutException`,`asyncio.wait_for` 抛 `asyncio.TimeoutError`,两者类型完全不同,抓错 = 等于没写 → gather 会崩。**取消机制三铁律**:(1) cancel 后必须 await task 让清理生效 (2) 抓 `CancelledError` 必须 raise,吞了 = 取消失效 task 继续跑 (3) Python 3.8+ `CancelledError` 继承 `BaseException` 防被 `except Exception:` 误吞。**Queue + 背压**:生产消费解耦,`maxsize` 逼慢生产者,消费者速度决定整体吞吐——加生产者没用,加消费者才提速。Q6 告警流作业 round-robin 输出完美印证。**踩坑集锦**:(1) 又忘加 await —— `asyncio.wait_for(...)` 前漏 await,r 变成协程对象调 status_code 崩(第三次踩同款坑);(2) `time.sleep` 用在异步里:同步阻塞 + `await time.sleep` TypeError(time 只留给 perf_counter 计时);(3) `task_done` 拼错两次 —— 先是 `taskdown` 后是 `task_down` —— done 不是 down;(4) 后台 task 异常静默丢失:consumer 崩了 main 不知道,`queue.join()` 死等 → 死锁卡 3 行后永不返回;工程防护是每个 task 外包 `try/except` 暴露错误。附带搞清三个词:**下游**=我依赖的服务(OnCall 里是 LLM/Loki/Prom/MySQL),**线程池**=预创建的 worker 线程避免重建开销(aiofiles 底层就是线程池),**503**=服务端临时不可用(通常因过载/下游挂/线程池满)。**解耦**概念:生产消费之间插一个 Queue,加消费通道不用改生产者 —— "改 A 不被迫改 B" 是工程目标。

### Day 12 - 并发对比
📅 完成日期:2026-04-28
📝 笔记:**GIL 本质 = 同一时刻一个 Python 进程里只允许一个线程跑字节码**;但锁的只是"执行 Python 字节码"这件事,IO 等待 / C 扩展计算时**会主动释放 GIL**——所以 threading 跑 IO 有效(等 IO 时让出锁),跑纯 Python CPU 完全无效(GIL 一直霸着,4 线程 ≈ 1 线程)。multiprocessing 真并行(每进程独立解释器 + 独立 GIL),但代价大:进程启动 + IPC 序列化 + 内存翻倍 + 不可 pickle 对象(lambda/嵌套函数/闭包)不能传。asyncio 单线程协作式,**没有 GIL 竞争问题**(只有一个线程在持锁),IO 场景比 threading 还快;但 CPU 任务下 `async def` 函数体没 `await` **就是普通同步函数**,套 async 不会自动并发。**选型口诀**:IO + 库支持 async → asyncio;IO + 同步库 → threading(或 `asyncio.to_thread` 包);CPU → multiprocessing。**实测数据印证 4 条反直觉认知**:(1)threading IO 0.51s vs multiprocess IO 1.87s —— 进程启动开销让多进程跑 IO 反而**慢 3.6 倍**;(2)threading CPU 2.36s ≈ 串行 2.39s —— **GIL 让多线程完全没并发**;(3)asyncio CPU 2.31s ≈ 串行 —— async def 没 await 单线程顺序跑;(4)multiprocess CPU 1.16s —— 真并行但 ≠ 1/N(进程启动 + IPC 摊不掉)。**踩坑**:函数调用没传参 / `return SyntaxError`(把内置异常类当占位符) / `pool.map` 是惰性的不 list 不阻塞 print 在 with 内立刻打印测出 0 秒 / CPU 任务传 `[2]*8` 没负载 / 用 `with httpx.AsyncClient():` 跑 asyncio.gather(完全用错且不需要)/ `asyncio.gather(*(coro * 20))` coroutine 不能 *20 / `asyncio.gather(...)` 同步顶层裸调要 asyncio.run 包 / `asyncio.to_thread(requests.get())` **加括号**导致同步先调用了再传(传函数对象 vs 传调用结果) / 8 个 print 标签 IO/CPU copy-paste 错 / Windows GBK 编码不认 emoji 要 PYTHONIOENCODING=utf-8 / multiprocess Windows 必须 `if __name__ == "__main__":` 守卫。/simplify 抓 11 处:抽 `@contextmanager def timed(label)` 干掉 8 段计时模板 / `cpu_task_async` 改 `return cpu_task(n)` 消除字节级重复 / 所有测试归到一个 `async def main()`(sync/async 流程合一)/ 常量提取(N_IO/N_CPU/CPU_LOAD)/ `[2]*20` → `range(N_IO)` / 多 with 链式让 pool 创建销毁也算入计时 / imports 合并 / `_async` 后缀替代数字后缀 / 删 8 处 WHAT 注释 / label 格式化对齐。**附带工程认知**:`asyncio.to_thread(func, *args)` 是 asyncio 和 threading 的正确合作姿势(老同步 SDK 混进 FastAPI 异步路由必用);"传函数对象不传调用结果"这条规律遍布 Python 所有"把函数交给别人执行"的 API(Thread/submit/装饰器/map/回调/Depends)。
