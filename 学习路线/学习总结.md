# 阶段一 · Python 高级语法

## Week 1 · 语法核心

### Day 1 · 装饰器基础(2026-04-18)

🎯 装饰器 = 吃函数吐函数的高阶函数。`@timing` ≡ `slow = timing(slow)`

📌 三层骨架

| 层 | 写法 | 干什么 |
|---|---|---|
| 外 | `def timing(func)` | 接被装饰函数 |
| 中 | `def wrapper(*a, **kw)` | 透传参数的替身 |
| 内 | `result = func(*a, **kw)` | 调一次原函数 |

💡 核心规则
1. wrapper 必须 `return result`,否则原返回值被吃掉变 `None`
2. `@functools.wraps(func)` 保 `__name__` / `__doc__`
3. `*args, **kw` 是签名透传万能模板,所有装饰器都这么写
4. 计时一律 `time.perf_counter()`(单调时钟),不用 `time.time()`

⚠️ 坑
- wrapper 忘 return → 拿到 None
- `wraps` 拼成 `wrap` → AttributeError
- `time.time()` 受系统时间校正漂移干扰

📝 最小代码
```python
import functools, time

def timing(func):
    @functools.wraps(func)
    def wrapper(*args, **kwargs):
        t0 = time.perf_counter()
        result = func(*args, **kwargs)
        print(f"{func.__name__}: {(time.perf_counter()-t0)*1000:.2f}ms")
        return result
    return wrapper
```

🔑 **外吞内替、参数透传、return 别忘、wraps 保身**

### Day 2 · 装饰器进阶(2026-04-19)

🎯 带参装饰器 = 三层嵌套(吃参数→吃函数→执行);类装饰器 = `__init__` 存状态 + `__call__` 执行

📌 几种形态对比

| 形态 | 等价展开 | 骨架嵌套 |
|---|---|---|
| `@timing` | `f = timing(f)` | 2 层 |
| `@retry(times=3)` | `f = retry(3)(f)` | 3 层 |
| `@Counter` | `f = Counter(f)` | 类:`__init__` 接 func |
| `@Retry(times=3)` | `f = Retry(3)(f)` | 类:`__init__` 接配置, `__call__` 接 func |

💡 核心规则
1. **带参装饰器必须写括号**,哪怕全默认也要 `@retry()`
2. 类装饰器**模块加载时只装一次**,实例固定 → `count` 等状态会一路涨
3. 装饰器叠加 `@A @B def f` ≡ `f = A(B(f))`,**装从内到外,调从外到内进出洋葱**
4. `try/except` 不是 `if/else`:**try 默认走通,抛异常才跳 except**
5. 闭包**修改**外层变量必须 `nonlocal`,否则被当局部变量 → `UnboundLocalError`(可改用 `dict["n"]` 绕)

⚠️ 坑
- `@retry` 漏括号 → `TypeError: missing 1 required positional argument`(Python 不查语义类型,只查参数个数)
- `except Exception:` 把 bug 型异常也吞了,导致重试掩盖真错
- 闭包 `counter += 1` 不加 nonlocal → UnboundLocalError

📝 最小代码
```python
def retry(times, delay):
    def decorator(func):
        @functools.wraps(func)
        def wrapper(*args, **kw):
            for i in range(times):
                try: return func(*args, **kw)
                except Exception:
                    if i == times - 1: raise
                    time.sleep(delay)
        return wrapper
    return decorator
```

🔑 **多一层括号多一层嵌套;闭包改值必声明;洋葱叠装从内到外、调用从外到内**

💎 加分
- 类装饰器适合"需要暴露 `.reset()` / `.count` 接口"的场景,函数装饰器写不出
- `@A @B` 想最先看日志的写最上面
- `return` 一执行函数立即结束,后面是 unreachable code

### Day 3 · 生成器 + yield(2026-04-19)

🎯 generator = **可暂停可恢复的函数**。函数体里出现 `yield` 就自动变 generator,调用不执行只返回对象,`next()` 才推进到下个 yield 冻结

📌 三大模式

| 模式 | 写法 | 用途 |
|---|---|---|
| 单流 | `def g(): yield x` | 流式吐数据 |
| 管道 | `g2(g1(...))` | 数据"流"过不落地 |
| 透传 | `yield from sub()` | 整批转发 |

💡 核心规则
1. **函数里有 yield = generator**,没有 yield = 普通函数(只写 print 不写 yield → 不会吐任何东西)
2. **yield 是暂停键,不是终止键** → `yield x; do_more()` 的 `do_more` 要下次 next 才跑
3. **generator 只能遍历一次**,用完即废,要重来必须重新调函数拿新对象
4. `yield from` 适合"整批透传",过滤/变换场景用 `for+if+yield`,**别乱用**
5. **不能 `len(generator)`**,数个数用 `sum(1 for _ in g)`

⚠️ 坑
- `yield line; line.rstrip()` → rstrip 对已吐出的值无效(yield 已暂停)
- `line.rstrip("\n")` 单独一行不接返回值 → str 不可变,白跑(写 `yield line.rstrip("\n")`)
- `for x in 3` 报错 → int 不可迭代

📝 最小代码
```python
def read_lines(path):
    with open(path) as f:
        for line in f:
            yield line.rstrip("\n")

def filter_keyword(lines, kw):
    for line in lines:
        if kw in line:
            yield line

# 管道:数据流不落地
for hit in filter_keyword(read_lines("a.log"), "ERROR"):
    print(hit)
```

🔑 **yield 是暂停键、yield from 是转发键、generator 是管道里一节水管**

💎 加分
- `for x in g` ≡ `it = iter(g); while True: try x=next(it) except StopIteration: break`
- `yield from sub()` 能接收 sub 的 `return` 值(在 `StopIteration.value` 里),协程双向通信前身

### Day 4 · 迭代器协议(2026-04-19)

🎯 `__iter__` + `__next__` + `StopIteration` = 可被 `for` 遍历的合同

📌 Iterable vs Iterator 核心区分

| | Iterable(数据本体) | Iterator(一次性游标) |
|---|---|---|
| 实现 | `__iter__` | `__iter__` + `__next__` |
| 能 `for` | ✅ | ✅ |
| 能 `next()` | ❌ | ✅ |
| 多次遍历 | ✅(每次新游标) | ❌ 用完即废 |
| 例子 | list / tuple / dict / str | generator / iter(lst) / 文件对象 |

💡 核心规则
1. **`iter(x) is x` 为 True ⟺ x 是 Iterator** —— 自己写 Iterator 时 `__iter__` 必须 `return self`
2. **class Iterator 的 `__next__` 不写循环**(Python 自动反复调)
3. **generator 必须写 `while True`**(否则跑到底就 StopIteration)—— class 不写循环、generator 必写循环,**位置正好相反**
4. generator 是 Iterator 协议的语法糖,Python 自动帮你实现协议

⚠️ 坑
- `next([1,2,3])` 炸 → list 是 Iterable 不是 Iterator,要先 `iter(...)`
- class `__next__` 里写 `while True` → 死循环
- 一个 Iterator 跑两遍 for → 第二遍空,要重新拿对象

📝 最小代码
```python
class Fib:
    def __init__(self):
        self.a, self.b = 0, 1
    def __iter__(self):
        return self                               # ⚠ Iterator 必须返回 self
    def __next__(self):
        v = self.a
        self.a, self.b = self.b, self.a + self.b
        return v
```

🔑 **Iterable 是书,Iterator 是书签;class 不写循环,generator 必写循环**

💎 加分
- 简单吐序列用 generator;需要 `.reset()` 接口用 class Iterator;想多次遍历用 class Iterable(`__iter__` 返回新 Iterator,像 list)

### Day 5 · 上下文管理器(2026-04-19)

🎯 `with` = **try/finally 的语法糖**,专门解决"拿了必须还"的资源场景(文件 / 连接 / 锁)

📌 两种实现对比

| 维度 | class 版 | `@contextmanager` 版 |
|---|---|---|
| `as xxx` 赋什么 | `__enter__` 返回值 | yield 后的表达式 |
| 感知异常 | `__exit__` 的 `exc_type` 参数 | `try/except` 在 yield 处捕获 |
| 异常向上抛 | 默认 `return None` 自动抛 | `except` 必须显式 `raise` |
| 清理代码 | `__exit__` 方法体 | `finally` 块 |

💡 核心规则
1. `__exit__` 默认 `return None` → 异常向上抛(我们要的);**`return True` 会吞异常**(慎用)
2. `@contextmanager` 的 yield **必须包 try/finally**,否则异常路径下清理全漏
3. `except` 里**忘 `raise` → 异常被静默吞**(看起来正常实际已炸)
4. 选型:简单进出用 `@contextmanager`(代码省一半);要细分异常类型 / 需要跨方法共享状态用 class

⚠️ 坑
- dunder 拼错 Python 不报错(`__intit__` 变普通方法,实例化时 `self.x` 没设)
- 裸 `except:` 吞 Ctrl-C(用 `except Exception:`)
- f-string `{}` 里只放变量,其他都是字面字符(不要写 `f"{exec: <sql>}"`)

📝 最小代码
```python
from contextlib import contextmanager

@contextmanager
def db_session():
    conn = connect()
    try:
        yield conn
        conn.commit()
    except Exception:
        conn.rollback()
        raise                # ⚠ 必须显式 raise
    finally:
        conn.close()
```

🔑 **with = 自动 try/finally;class 默认透传异常,contextmanager 要显式 raise**

### Day 6 · 魔法方法 + 动态属性(2026-04-20)

🎯 dunder 方法 = Python 语法糖的接插件,让自定义类融入 `print` / `==` / `()` / `.` 原生语法

📌 5 个核心方法

| 方法 | 触发时机 | 干什么 |
|---|---|---|
| `__init__` | `Foo(...)` | 实例化 |
| `__call__` | `obj(...)` | 实例当函数调 |
| `__repr__` | `print(obj)` / f-string / list 显示 | 调试字符串 |
| `__eq__` | `obj == x` | 自定义相等 |
| `__getattr__` | `obj.x` **正常查找失败**的兜底 | 动态属性 / 点取 dict |

💡 核心规则
1. `__getattr__` 是查找**失败**的兜底,不是每次访问都触发(那是 `__getattribute__`)
2. `__eq__` 类型不匹配 **`return NotImplemented`** 不是 `return False`(让对方 `__eq__` 试,保对称性)
3. **重写 `__eq__` 后 `__hash__` 自动变 None** → 对象不能进 set / dict key,要支持就同时重写 `__hash__`(用同一组字段)
4. EAFP 优于 LBYL:`try: d[k] except KeyError` 比 `if k in d: d[k]` 命中只查 1 次
5. `f"{v!r}"` 调 `repr(v)`(字符串带引号),嵌套对象自动递归展开

⚠️ 坑
- `__getattr__` 里访问 `self._data`,若 `_data` 未初始化会**无限递归**(防御:第一行守 `if k == "_data": raise AttributeError(k)`)
- `if v:` 把 0 / False / "" / [] 误判为"不存在",要用 `if k in d`
- `.values` 漏括号变方法对象,永远不等

📝 最小代码
```python
class Config:
    def __init__(self, data):
        self._data = {k: (Config(v) if isinstance(v, dict) else v) for k, v in data.items()}
    def __getattr__(self, k):
        try: return self._data[k]
        except KeyError:
            raise AttributeError(f"{type(self).__name__!r} has no attr {k!r}") from None
    def __repr__(self):
        return f"Config({', '.join(f'{k}={v!r}' for k, v in self._data.items())})"
    def __eq__(self, other):
        if not isinstance(other, Config): return NotImplemented
        return self._data == other._data
```

🔑 **getattr 是查找兜底、eq 不匹配返 NotImplemented、重写 eq 别忘 hash**

💎 现实对应:Pydantic `BaseModel` / OmegaConf / HuggingFace `AutoConfig` / `argparse.Namespace` 底层都是这套

### Day 7 · Week 1 综合复盘(2026-04-21)

🎯 综合作业 `tinylog` = 装饰器 + 上下文管理器 + 生成器三件套同时激活,验证 Week 1 单点能否拼成整体

📌 三件套各管什么

| 机制 | 角色 | 在 tinylog 里 |
|---|---|---|
| 装饰器 | 函数包装 | `@log_calls` 装饰处理函数 |
| 上下文管理器 | 带清理的代码块 | `with log_block("批次")` |
| 生成器 | 流式吐数据 | `tail(path)` 行流 |

💡 核心规则
1. 装饰器 `start = time.perf_counter()` **必须放 wrapper 内**,放外层只算装饰那一刻
2. **`finally: return` 会吞异常** —— 异常路径用 `except: ...; raise`,不用 finally
3. **func 在装饰器里只能调一次**,`result = func(...)` 存起来 → 副作用函数(发邮件 / 转账)双调 = 生产事故
4. **重写 `__eq__` 自动让 `__hash__ = None`** → 对象不能进 set,要支持要同时写 `__hash__`(用同一组字段)
5. **类型注解运行时不做检查**,只服务 IDE / mypy / Pydantic / FastAPI 这种主动读注解的库

⚠️ 坑
- `func.__name__(*args)` 把字符串当函数调 → TypeError(`__name__` 是属性不是方法)
- `except Exception: raise` 是无用模式 = 没抓
- `tail` 把 `with open` 放生成器内 → 文件什么时候关取决于生成器**耗尽 / GC / .close()**,不是 caller for 结束

📝 最小代码
```python
def log_calls(func):
    @functools.wraps(func)
    def wrapper(*args, **kw):
        start = time.perf_counter()                # ⚠ 在 wrapper 里
        try:
            result = func(*args, **kw)             # ⚠ 只调一次
        except Exception as e:
            elapsed = time.perf_counter() - start
            print(f"!!! {func.__name__} {type(e).__name__}: {e} after {elapsed:.4f}s")
            raise                                  # ⚠ 重抛
        elapsed = time.perf_counter() - start
        print(f"<<< {func.__name__} → {result!r} in {elapsed:.4f}s")
        return result
    return wrapper
```

🔑 **start 在 wrapper 内、func 只调一次、finally 别 return、异常该抛就抛**

💎 类型注解
- `list[int]` / `dict[str, int]`(3.9+ 内建)优先于 `typing.List`
- `int | str`(3.10+)替代 `Union[int, str]`,`str | None` 替代 `Optional[str]`

## Week 2 · 异步 + 工程化

### Day 8 · asyncio 基础(2026-04-21)

🎯 单线程并发:协程等 I/O 时让出 CPU 给别人,**异步治"等待"不治"算得慢"**

📌 四件套

| 关键字 / API | 作用 |
|---|---|
| `async def foo()` | 定义协程函数。**调用不执行**,只返回协程对象 |
| `await x` | 等 x 完成,期间让出 CPU。只能在 async def 里用 |
| `asyncio.run(协程)` | 启动事件循环,程序通常只调一次 |
| `asyncio.gather(*coros)` | 并发跑多个,等全部完成,**按传入顺序**返回结果 |

💡 核心规则
1. **判用不用异步看任务**:I/O 密集(爬虫 / LLM / DB)✅,CPU 密集(矩阵 / 加密)❌ 该上多进程
2. **顺序 `await` 多个协程 = 串行**,光写 async 不加速,真正并发靠 `gather` / `create_task`
3. **`gather(coros)` ≠ `gather(*coros)`** —— gather 要的是多个位置参数,不是一个容器
4. 异步代码里**不能混同步 I/O**(`httpx.get` 阻塞整个事件循环)
5. `httpx.AsyncClient` 是类要实例化:`async with httpx.AsyncClient() as c`,不是 `async with httpx.AsyncClient as c`

⚠️ 坑
- `async def foo()` 调用不 await → `coroutine 'foo' was never awaited` 警告(头号坑)
- 每个请求开新 client → 重建连接池,十请求十套握手(共享 client 提速 41%)
- `g([1,2,3])`(`def g(*args)`)→ args 是 `([1,2,3],)`,不是 `(1,2,3)`(调用处 `*` 才是拆包)

📝 最小代码
```python
import asyncio, httpx

async def fetch(client, url):
    r = await client.get(url)
    return r.status_code

async def main():
    urls = ["https://example.com"] * 10
    async with httpx.AsyncClient() as client:        # 共享 client
        results = await asyncio.gather(*(fetch(client, u) for u in urls))
    print(results)

asyncio.run(main())
```

🔑 **async 不加速 await 才让出、gather 必须解包、共享 client 别重建**

💎 `*` / `**` 速记
- 定义时打包:`def f(*args)` → tuple,`def f(**kw)` → dict
- 调用时拆包:`f(*lst)` 等价 `f(1,2,3)`,`f(**d)` 等价 `f(a=1, b=2)`
- 一句话:**定义时合,调用时拆**

### Day 9 · async 深入:Task / Semaphore / wait_for / cancel / Queue(2026-04-21)

🎯 `gather` 是发动机,Day 9 学的是控制发动机的五个旋钮

📌 五旋钮

| 旋钮 | 作用 | 何时用 |
|---|---|---|
| `create_task` | 起步发车,可先发后等 | fire-and-forget(心跳 / LLM 流式) |
| `Semaphore(N)` | 同时最多 N 个 | 控并发数(防 429 / 封 IP) |
| `wait_for(coro, t)` | 超时自动 cancel | 防下游卡死拖垮自己 |
| `task.cancel()` | 注入 CancelledError | 用户中断 / 主程序关后台 |
| `Queue(maxsize)` | 生产消费解耦 + 背压 | 速度不匹配场景 |

💡 核心规则
1. **Semaphore 甜点公式**:`并发 ≈ rate × 平均耗时 × 0.5`(留 buffer)
   - OpenAI 500rpm × 3s ≈ 10-20;本地 1000 文件 ≈ 50-100;MySQL 池 20 ≈ 15
2. **超时异常类型谁管谁抓**:`httpx.AsyncClient(timeout=3)` → `httpx.TimeoutException`;`asyncio.wait_for` → `asyncio.TimeoutError`,**抓错 = 没抓**
3. **取消三铁律**:cancel 后必须 await task / 抓 CancelledError 必须 raise / 用 `except Exception` 而不是裸 except 防误吞(3.8+ 已让 CancelledError 继承 BaseException)
4. **Queue 速度真理**:加生产者不提速,**加消费者才提速**;系统稳态吞吐 = 消费者数 / 单消费耗时
5. `await t1` 期间所有 task 都在跑,**完成顺序只看各自耗时**,跟 await 谁无关

⚠️ 坑
- 后台 task 异常**静默丢失**:consumer 崩了 main 不知道,`queue.join()` 死等 → 死锁(防护:每个 task 外包 try/except 暴露错误)
- `time.sleep` 用在异步里 → 同步阻塞整个事件循环;`await time.sleep` → TypeError(`time` 只留给 `perf_counter`)
- `task_done` 拼成 `task_down` —— done 是完成,不是下去

📝 最小代码
```python
async def fetch(client, sem, url):
    async with sem:                                   # 令牌:用了自动还
        try:
            r = await asyncio.wait_for(client.get(url), timeout=3.0)
            return r.status_code
        except (asyncio.TimeoutError, httpx.TimeoutException):
            return -1                                  # 抓两种类型

async def main():
    sem = asyncio.Semaphore(3)
    async with httpx.AsyncClient() as client:
        results = await asyncio.gather(*(fetch(client, sem, u) for u in urls))
```

🔑 **Semaphore 防炸下游、wait_for 防被下游炸、cancel 三铁律不能破、Queue 加消费者才提速**

💎 工程词汇
- **下游 / 上游**:谁调我 = 上游(用户 / 前端),我调谁 = 下游(LLM / DB)。"下游超时 / 限流 / 抖动"
- **线程池**:预创建 worker 线程池,asyncio 默认 = `min(32, CPU+4)`;`aiofiles` 底层就是它
- **HTTP 429 / 503**:OnCall 最常见两个告警;429 = 限流(客户端锅),503 = 过载(服务端锅)
- **解耦**:改 A 不被迫改 B;Queue 是最典型解耦工具

### Day 10 · 异步生态(2026-04-27)

🎯 把 Day 9 的"异步骨架"(gather / Semaphore / wait_for)落到真实工程:`httpx.AsyncClient` + `aiofiles` + 失败重试

📌 关键 API

| API | 干什么 |
|---|---|
| `httpx.AsyncClient(timeout=, limits=)` | 异步 HTTP,内部连接池 |
| `httpx.Limits(max_connections=, max_keepalive_connections=, keepalive_expiry=)` | 限 TCP 连接数(传输层) |
| `httpx.Timeout(connect=, read=, write=, pool=)` | 4 档超时拆分 |
| `aiofiles.open(path, mode)` | 异步文件 IO(线程池包装) |
| `r.raise_for_status()` | 4xx/5xx 抛 `HTTPStatusError` |
| `r.content` / `r.text` / `r.json()` | bytes / str / dict |

💡 核心规则
1. **`max_connections` 限传输层、`Semaphore` 限应用层**,颗粒度不同可叠加(谁小谁先生效)
2. **`aiofiles` 不是真异步磁盘 IO** —— 本质 `loop.run_in_executor` 丢线程池,目的是不阻塞事件循环
3. **超时类型谁管谁抓**:`httpx.TimeoutException`(httpx 自抛)≠ `asyncio.TimeoutError`(`wait_for` 抛),不要既设 client timeout 又套 wait_for
4. **`Semaphore` 包 retry vs 包单次的 tradeoff**:包 retry → sleep 期间占槽,实际并发 <10;包单次 → 失败方立刻又来抢,生产要加 jitter 防雷暴
5. **FastAPI 路由选型**:`async def` 进事件循环(里面**绝不能写 sync IO**);`def` 自动进线程池(里面写 sync 代码 OK)

⚠️ 典型坑
- `asyncio.sleep(1)` 漏 await → 协程对象被丢弃,根本没 sleep
- `"ab"`(append)模式重试 → 半段坏数据 + 完整数据拼一起,损坏文件;下载场景一律 `"wb"`
- f-string 里 `"DOWNLOAD_DIR/..."` 没 `{}` 包变量 → 字面字符串"DOWNLOAD_DIR",写到不存在的目录直接 `FileNotFoundError`

📝 最小代码
```python
import asyncio, httpx, aiofiles
from pathlib import Path

DOWNLOAD_DIR = Path(__file__).parent / "downloads"
MAX_RETRIES = 2

async def download(client, url, sem, i):
    async with sem:
        for attempt in range(MAX_RETRIES + 1):
            try:
                r = await client.get(url)
                r.raise_for_status()
                async with aiofiles.open(DOWNLOAD_DIR / f"file_{i}.bin", "wb") as f:
                    await f.write(r.content)
                return
            except httpx.TimeoutException:
                if attempt < MAX_RETRIES:
                    await asyncio.sleep(1)
                else:
                    raise

async def main():
    DOWNLOAD_DIR.mkdir(exist_ok=True)
    sem = asyncio.Semaphore(10)
    async with httpx.AsyncClient(timeout=10) as client:
        results = await asyncio.gather(
            *(download(client, url, sem, i) for i, url in enumerate(urls)),
            return_exceptions=True,
        )

if __name__ == "__main__":
    asyncio.run(main())
```

🔑 **httpx 限连接、aiofiles 不阻事件循环、超时类型谁管谁抓、retry 包不包 sem 是 tradeoff**

💎 加分
- `httpbin.org/bytes/{n}` 是公共测试 URL,返回 n 字节随机 bytes(国内不稳,timeout 设 10s 比较稳)
- `r.raise_for_status()` 别忘加,否则 5xx 当成功写入文件,统计造假
- 写文件路径用 `Path(__file__).parent / "..."`,绝对路径不依赖 cwd

### Day 11 · 类型 + Pydantic v2(2026-04-28)

🎯 type hint 默认是给「人 + IDE + 静态检查」看的,**Python 解释器运行时不管**;Pydantic 借用 type hint 语法,**在运行时真的强制校验**

📌 typing 新老写法

| 概念 | 老 | 新(推荐) |
|---|---|---|
| int 列表 | `List[int]` | `list[int]` |
| str→int 字典 | `Dict[str, int]` | `dict[str, int]` |
| 可空 | `Optional[str]` | `str \| None` |
| 多选一 | `Union[int, str]` | `int \| str` |
| 枚举值 | - | `Literal["a", "b"]` |
| 字典形状 | - | `TypedDict`(运行时不强制) |
| 鸭子接口 | - | `Protocol` |

📌 Pydantic v2 核心 API

| 工具 | 用途 |
|---|---|
| `BaseModel` | 继承得到带运行时校验的数据类 |
| `Field(default=, min_length=, ge=, ...)` | 字段约束(default 一律前置) |
| `model_config = ConfigDict(extra="forbid", str_strip_whitespace=True)` | 模型级配置(写在类顶部) |
| `@field_validator("x") @classmethod def fn(cls, v)` | 单字段自定义校验,return v |
| `@model_validator(mode="after") def fn(self)` | 跨字段校验(self 已构造好),return self |
| `model_dump()` / `model_validate()` | dict 互转(替代 v1 `.dict()` / `parse_obj()`) |

💡 核心规则
1. **`x: int \| None` 仍是必填**!要可省略必须加 `= None` 或 `Field(default=None, ...)`(头号坑)
2. **Literal 来自 typing,不在 pydantic 里**
3. **校验顺序**:`str_strip_whitespace` 先 strip → Field 约束 → field_validator;**前面能拦的就别写 validator**
4. **可变默认必须 `default_factory=dict / list`**,直接 `= {}` v2 会报错
5. **v1 → v2 关键变更**:`.dict()` → `model_dump()` / `class Config:` → `model_config = ConfigDict(...)` / `parse_obj()` → `model_validate()`(网上老教程坑多)

⚠️ 坑
- `name: str | None = Field(min_length=1, max_length=50)` 不加 default → name **必填**(Optional 类型 ≠ 可选字段)
- `BaseModel` / `Field` / `ConfigDict` 拼写错 IDE 会标红一定要扫(Basemodel / Filed)
- `temperature: float = Field(gt=0.0, lt=2.0)` 把 0 排除 → LLM API 中 0 是合法值(贪心采样),要 `ge / le`
- `model_` 前缀字段(`model_name`)撞 Pydantic 内置命名空间警告 → `protected_namespaces=()` 关掉

📝 最小代码
```python
from pydantic import BaseModel, Field, ConfigDict, model_validator, ValidationError
from typing import Literal


class Message(BaseModel):
    model_config = ConfigDict(extra="forbid", str_strip_whitespace=True)

    role: Literal["user", "assistant", "system"]
    content: str = Field(min_length=1, max_length=4000)
    metadata: dict[str, str] = Field(default_factory=dict)
    name: str | None = Field(default=None, min_length=1, max_length=50)

    @model_validator(mode="after")
    def system_role_forbids_name(self):
        if self.role == "system" and self.name is not None:
            raise ValueError("role=system 时 name 必须为 None")
        return self
```

🔑 **type hint 默认软约束、Pydantic 把它变硬约束;Optional 类型不是可选字段,要 default;Literal 在 typing 不在 pydantic;mutable 默认值用 default_factory**

💎 加分
- 实战里 `extra="forbid"` 强烈推荐 —— LLM 输出多余字段时立刻知道,而不是悄悄丢掉
- 真实场景 90% 用 `mode="after"` 的 model_validator(self 已构造好,操作直观);`mode="before"` 拿到原始 dict 用来"预处理输入"(少用)
- 校验函数命名要描述规则陈述,不要叫 `validate1` / `check`(出错堆栈一眼能懂)
- v1 → v2 看到 `.dict()` / `class Config:` 这些就要警觉是老代码

### Day 12 · 并发对比:GIL / threading / multiprocessing / asyncio(2026-04-28)

🎯 **GIL = 同一时刻一个 Python 进程里只允许一个线程跑字节码**;选型按任务类型 + 是否有 async 库

📌 三方式适用场景

| 维度 | threading | multiprocessing | asyncio |
|---|---|---|---|
| 真并行 | ❌(GIL) | ✅(每进程一把 GIL) | ❌(单线程) |
| IO 加速 | ✅ | ✅(代价大) | ✅✅ 最优 |
| CPU 加速 | ❌ | ✅ 唯一选择 | ❌ |
| 资源开销 | 中(每线程 ~1MB) | 高(完整解释器) | 极低(每协程 KB) |
| 开几个 | 几百 | ≤ CPU 核数 | 几千~几万 |
| 数据传递 | 共享内存 | **要 pickle** | 共享内存 |
| 切换成本 | 微秒级(内核) | - | 纳秒级(用户态) |
| 切换时机 | 抢占式(OS) | - | 协作式(必须 await) |

💡 核心规则
1. **GIL 锁的是 Python 字节码执行**,IO 等待 / C 扩展计算时**会释放**(所以 threading 跑 IO 有效,跑纯 Python CPU 无效)
2. **asyncio 单线程没有 GIL 竞争问题**(只有一个线程在持锁),纯 IO 场景比 threading 还快
3. **`async def` 函数体里没 `await` 就是普通同步函数** —— 套 async 不会自动并发
4. **Windows 下 multiprocessing 必须 `if __name__ == "__main__":` 守卫**(spawn 模式重新 import 主模块)
5. **传函数对象不传调用结果**:`asyncio.to_thread(func, *args)` ✅,`asyncio.to_thread(func())` ❌(同 Thread/submit/装饰器/map/回调,所有"把函数交给别人执行"的 API 同规律)
6. **选型口诀**:IO + async 库 → asyncio;IO + 同步库 → threading(或 `asyncio.to_thread` 包);CPU → multiprocessing

⚠️ 坑
- `pool.map(...)` **是惰性的**,不 `list()` 强制迭代不会等任务完成 → 测耗时全是 0
- `asyncio.gather(*(coro * 20))` —— coroutine 不能 `*20`,要 `[func(i) for i in range(20)]`
- `asyncio.gather(...)` 在同步顶层裸调 → 必须 `asyncio.run(main())` 包
- `asyncio.to_thread(requests.get())` —— 括号一加,同步先调用了再传给线程池,等于没异步
- Windows 终端默认 GBK 不认 emoji → `PYTHONIOENCODING=utf-8` 临时解或系统环境变量永久解

📝 最小代码
```python
import asyncio, os, time
from concurrent.futures import ProcessPoolExecutor, ThreadPoolExecutor
from contextlib import contextmanager

@contextmanager
def timed(label):
    start = time.perf_counter()
    yield
    print(f"{label:<14} {time.perf_counter() - start:.3f}s")

async def main():
    # IO 密集
    with timed("IO threading"), ThreadPoolExecutor(max_workers=20) as pool:
        list(pool.map(io_task, range(20)))
    with timed("IO asyncio"):
        await asyncio.gather(*[io_task_async(i) for i in range(20)])

    # CPU 密集
    with timed("CPU multi"), ProcessPoolExecutor(max_workers=os.cpu_count()) as pool:
        list(pool.map(cpu_task, [5_000_000] * 8))

if __name__ == "__main__":
    asyncio.run(main())
```

🔑 **GIL 卡死多线程跑 CPU、async 没 await 等于串行、IO 选 asyncio/threading、CPU 必须 multiprocess**

💎 实测数据印证 4 条反直觉(本机 Windows)
- threading IO 0.51s vs multiprocess IO **1.87s** —— 进程启动开销让多进程跑 IO 反而慢 3.6 倍
- threading CPU **2.36s ≈ 串行 2.4s** —— GIL 让多线程完全没并发
- asyncio CPU **2.5s ≈ 串行** —— async def 函数体没 await,单线程顺序跑
- multiprocess CPU 1.16s —— 真并行但 ≠ 1/N(进程启动 + IPC 摊不掉)

💎 工程姿势
- **`asyncio.to_thread(func, *args)`** 是 asyncio 和 threading 的正确合作:老同步 SDK 混进 FastAPI 异步路由必用,防止阻塞事件循环
- **`with timed(...), Pool() as pool:`** 多 with 链式,`__enter__` 顺序进、`__exit__` 反序出,pool 创建销毁也被 timed 包住

```python
async def check_all(urls, max_concurrent=5, timeout=5.0):

    async with httpx.AsyncClient(follow_redirects=True) as client:
        async def _bounded(url):                     # ⭐ 每 task 内部 acquire
            async with sem:
                return await check_url(client, url, timeout=timeout)
        return await asyncio.gather(*(_bounded(u) for u in urls))
```

### Day 15 · FastAPI 路由基础(2026-05-04)

🎯 web 后端 = 一个 `while True: req = recv(); resp = handle(req); send(resp)` 的死循环;FastAPI = 把"路由表 + 类型注解 + Pydantic 校验"自动化的框架。

📌 三类参数 / 三种来源

| 来源 | 写法 | 哪来的 |
|---|---|---|
| **Path Parameter** | `@app.get("/items/{item_id}")` + `def f(item_id: int)` | URL 路径里 `{xxx}` 占位 |
| **Query Parameter** | `def f(q: str, limit: int = 10)` | URL `?q=xx&limit=5` 里 |
| **Request Body** | `def f(todo: TodoCreate)` 接 Pydantic Model | POST/PUT 的 body(JSON) |

**规则**:函数参数没出现在 path 里 + 是基础类型 → Query;是 BaseModel 子类 → Body。

💡 核心规则
1. **类型注解即校验**:`item_id: int` → `/items/abc` 自动 422,带 `loc: ["path", "item_id"]` 精确定位错误位置
2. **Query 默认值控制必填**:有默认 = 可选;无默认 = 必填没传 → 422
3. **Body 用 Pydantic Schema**:Day 11 学的 `Field(min_length=1, ge=1, le=5)` 直接当 API 校验,**不写一行 if/else**
4. **HTTPException 主动报 4xx**:`raise HTTPException(status_code=404, detail=f"...")` —— GET/PUT/DELETE 单个对象前必须先存在性检查
5. **REST 路径用名词不用动词**:`GET /todos`(列出)/ `POST /todos`(创建)/ `GET /todos/{id}`(单个) —— 同 path 不同 method = 不同动作;嵌套表达层级 `/articles/5/comments/42`
6. **状态码契约**:201(POST 创建成功)/ 204(DELETE 成功无 body)/ 404(资源不存在)/ 422(校验失败,FastAPI 自动)
7. **`def` vs `async def` 路由**:无 IO 用 `def` 让 FastAPI 自动丢线程池;有 async IO(httpx / asyncpg)才用 `async def`,**绝不能在 async def 里写 sync IO**(阻塞 event loop)

⚠️ 坑
- `is not` ≠ `not in`:前者身份比较只用于 None,后者成员关系用于 dict/list/set —— 写错全部触发 404
- `.values` 漏括号(Day 6 同款坑复发):方法不带括号 = 拿到方法对象本身
- f-string 漏 `f` 前缀 → `{var}` 字面输出不替换
- shell 含 `&` 必须用引号包(`&` 是后台运行符号);PowerShell 的 `curl` 是 `Invoke-WebRequest` 别名不是真 curl
- 函数内修改全局变量必须 `global` 声明(读取不需要,写入才需要)

📝 最小代码 — 一个完整 CRUD 端点的范式

```python
def _get_or_404(todo_id: int) -> dict:
    todo = todos.get(todo_id)              # 一次查 (避免 TOCTOU 双查)
    if todo is None:
        raise HTTPException(status_code=404, detail=f"Todo {todo_id} 不存在")
    return todo

@app.put("/todos/{todo_id}")
def update_todo(todo_id: int, todo: TodoCreate):
    _get_or_404(todo_id)
    todos[todo_id] = {"id": todo_id, **todo.model_dump()}   # ⭐ Pydantic 解包,字段加减不动 CRUD 代码
    return todos[todo_id]
```

🔑 **类型注解即校验 + REST 用名词路径 + HTTPException 报 4xx + `_get_or_404` 模式 + `**model_dump()` 防字段重复**

💎 加分
- Swagger UI (`/docs`) 自动生成交互式文档 + curl 命令,改 schema 自动同步;ReDoc (`/redoc`) 是只读文档版
- 422 错误的 `loc` 数组是 `[来源, 字段名]`(`["path", "item_id"]` / `["query", "q"]` / `["body", "title"]`),前端能精确定位错误
- URL 含中文自动 percent encoding(`电子` → `%E7%94%B5%E5%AD%90`),FastAPI 自动解码,代码里收到的还是字符串
- 内存全局状态(`next_id += 1`)在多 worker 下竞态 → 生产用 DB autoincrement / UUID / Snowflake(面试常考)
- `_get_or_404` 助手 Day 16 升级成 `Depends`,自动出现在 OpenAPI 的 404 响应里

### Day 16 · Pydantic + 依赖注入(2026-05-06)

🎯 `response_model` 控制响应形状 + `Depends` 把前置逻辑变成可注入组件 + `responses=` 显式声明业务异常

📌 三套 API 用法

| 用途 | 写法 | 关键点 |
|---|---|---|
| 输出 schema | `response_model=TodoOut` | 白名单过滤字段 + OpenAPI 文档同步 |
| list 输出 | `response_model=list[TodoOut]` | 对每项过滤 |
| 取数据型依赖 | `existing: dict = Depends(get_todo_or_404)` | 拿返回值 |
| 门禁型依赖 | `dependencies=[Depends(verify_token)]` | 不要返回值,纯校验 |
| 业务异常文档 | `responses={**NOT_FOUND, **UNAUTHORIZED}` | dict 合并,不是 list |
| Header 取值 | `x_token: str = Header(...)` | 下划线↔横线自动转,`...` 表必填 |

💡 核心规则
1. **输入严校验,输出纯类型**:`TodoCreate` 写 `Field(min_length=, ge=...)`;`TodoOut` 只写 `id: int`,**不写默认值/约束** —— 服务端可信,默认值会蒙混 bug
2. **204 不带 body**:`status_code=204` 配 `response_model` 自相矛盾,只能保留状态码
3. **single source of truth**:有了 `existing = Depends(get_todo_or_404)`,端点里只用 `existing["id"]`,不要再写 `todo_id` —— 否则原始 path 和加工后结果可能不一致
4. **schema 层自动登记,运行时业务异常手动声明** —— 422 是参数类型注解推得出来的,**404/401/403 框架看不见你函数体里的 `raise`**,必须 `responses=` 显式声明
5. **依赖默认按请求级 cache**:同一次请求里同一个依赖只跑一次,key 是依赖函数本身
6. **互相独立的依赖不保证按声明顺序执行**:需要顺序就嵌套表达(`Depends(A)` 在 B 内部),不要靠"我把 A 写在 B 前面"

⚠️ 坑
- IDE 把 `int` 自动补全成 `InterruptedError` —— Pydantic schema 错误延迟到生成时才报,启动失败先扫所有字段类型
- `responses=[NOT_FOUND, UNAUTHORIZED]` 写成 list —— FastAPI 拿不到状态码 key,文档全没出现;正解 `{**NOT_FOUND, **UNAUTHORIZED}`
- `vertify_token` 拼写错(应 `verify`),前后一致不报错但 OpenAPI 暴露
- DELETE 写 `status_code=204` + `response_model` 自相矛盾
- 把 `get_todo_or_404` 写到 `dependencies=[]` 里 —— 返回值被丢,端点没数据可用,白查一次

📝 最小代码 — 一个完整的鉴权 + 取数据端点

```python
class TodoOut(BaseModel):                            # 输出:纯类型必填
    id: int; title: str; done: bool; priority: int

class ErrorDetail(BaseModel):
    detail: str

NOT_FOUND_RESPONSE = {404: {"model": ErrorDetail, "description": "Todo 不存在"}}
UNAUTHORIZED_RESPONSE = {401: {"model": ErrorDetail, "description": "操作未授权"}}

def get_todo_or_404(todo_id: int) -> dict:
    todo = todos.get(todo_id)
    if todo is None:
        raise HTTPException(404, f"Todo {todo_id} 不存在")
    return todo

def verify_token(x_token: str = Header(...)) -> None:
    if x_token != "secret123":
        raise HTTPException(401, "Invalid token")

@app.put(
    "/todos/{todo_id}",
    response_model=TodoOut,
    responses={**NOT_FOUND_RESPONSE, **UNAUTHORIZED_RESPONSE},   # 多响应合并
    dependencies=[Depends(verify_token)],                         # 门禁
)
def update_todo(
    todo_in: TodoCreate,
    existing: dict = Depends(get_todo_or_404),                    # 取数据
):
    todos[existing["id"]] = {"id": existing["id"], **todo_in.model_dump()}
    return todos[existing["id"]]
```

🔑 **输入严输出纯 + 取值当参数门禁进 list + 业务异常手动声 + 依赖给的拿来用别再读 path**

💎 加分
- `Depends(verify_token, use_cache=False)` 可关掉单请求 cache(罕见,如每次都要新 trace_id)
- 嵌套依赖:`get_current_user` 内部 `Depends(verify_token)`,FastAPI 拓扑解析整张图
- 错误响应字段 `detail` 是 FastAPI HTTPException 写死的;想自定义形状要写 exception handler(Day 17 学)
- `_` 前缀加不加,看的不是函数重要性,是"它打算被多远的代码调用" —— 只在本文件用 → 加;被其他模块装配/测试/mock → 不加
- Swagger UI 锁头图标:有 `dependencies=[Depends(verify_token)]` 的端点会显示 🔒(本质来自 OpenAPI security scheme,这里是简化版鉴权)
