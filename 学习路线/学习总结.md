#### 装饰器(Day 1 · 2026-04-18)

**是什么**
装饰器本质就是一个**高阶函数**:吃一个函数进来,吐一个"包装过的新函数"出去。`@timing` 写在 `def slow` 上面,只是语法糖,真正发生的事是:

```python
slow = timing(slow)   # @timing 等价于这一行
```

**为什么需要**
想给一堆已有函数"叠一层能力"(计时、重试、日志、鉴权),又不想去改每个原函数的代码。装饰器就是 Python 里"横切关注点"的标准解法 —— 功能代码和业务代码分开。

**核心用法(三层骨架)**

```python
import functools, time

def timing(func):                         # 第一层:接收被装饰的函数
    @functools.wraps(func)                # 保住原函数的 __name__ / __doc__
    def wrapper(*args, **kwargs):         # 第二层:真正替身,透传参数
        t0 = time.perf_counter()
        result = func(*args, **kwargs)    # 调一次原函数
        print(f"{func.__name__} 耗时 {(time.perf_counter()-t0)*1000:.2f} ms")
        return result                     # ⚠ 必须把结果 return 出去
    return wrapper                        # 第一层返回 wrapper 替换掉原函数

@timing
def add(a, b): return a + b
```

**典型坑(自己踩过的 3 个)**
1. **wrapper 忘 return 原函数结果** → 调用 `add(2,3)` 拿到的是 `None`,因为 wrapper 默认返回 `None`,原函数的返回值被吃掉了
2. **`functools.wraps` 拼成 `wrap`**(单数) → `AttributeError`,因为 `wrap` 是另一个不相关的东西
3. **计时用 `time.time()`** → 它是墙钟,会被系统时间校正跳动干扰。想测"代码跑了多久"一律用 `time.perf_counter()`,这是单调时钟,只增不减

**附:万能参数 `*args, **kwargs`**

- `*args`:把"位置参数"打包成元组。`def f(*args)`,调用 `f(1, 2, 3)` → `args = (1, 2, 3)`
- `**kwargs`:把"关键字参数"打包成字典。`def f(**kwargs)`,调用 `f(a=1, b=2)` → `kwargs = {"a":1, "b":2}`
- 在装饰器里必须写 `def wrapper(*args, **kwargs)` + `func(*args, **kwargs)` → **签名透传**:无论原函数长什么样(要几个参数、要不要关键字参数),wrapper 都能原样接住再原样喂回去
- `*` 和 `**` 出现在**函数定义**里是"打包",出现在**函数调用**里是"拆包":`f(*args)` 等价于把元组里的元素一个一个展开当位置参数传进去
- 记忆锚:**定义时打包成集合,调用时拆包发出去**

**一句话记忆**
装饰器 = "外层函数吞函数、内层 wrapper 替身干活、`*args/**kwargs` 透传参数、别忘了 return、元信息靠 wraps 保住"。

**延伸:带参装饰器(Day 2 · 2026-04-18)**

**是什么**
`@retry(times=3, delay=1)` 这种**带参**装饰器,本质是:

```python
hello = retry(times=3, delay=1)(hello)
```

先用参数调一次 `retry(...)` 得到"真装饰器",再用它包 hello。比 Day 1 的 `hello = timing(hello)` 多了一步,所以函数嵌套**从 2 层变 3 层**。

**为什么**
装饰器自己也要接收配置(重试几次、延迟多久),又不能塞进 wrapper 的参数(wrapper 要透传**被装饰函数**的参数)。于是把"接收装饰器参数"单独拎到最外一层。

**怎么用(三层骨架)**

```python
def retry(times, delay):           # 第1层:吃装饰器参数
    def decorator(func):           # 第2层:吃被装饰函数(= Day1 的 timing)
        def wrapper(*args, **kw):  # 第3层:实际执行
            ...
        return wrapper
    return decorator
```

**坑在哪(自己推过一次)**
`@retry` 漏写括号 **一定报错**,但报的不是"参数应该是函数",而是:

```
TypeError: retry() missing 1 required positional argument: 'delay'
```

`@retry`(无括号)展开成 `hello = retry(hello)`。Python **不检查**你传的是不是函数(它不在乎语义类型),只检查**参数个数**。`retry` 要 2 个参数,只给了 1 个(hello 被当成 `times`),所以缺参。

**一句话记忆**
装饰器有几个参数,就比普通装饰器多包一层;**带参装饰器必须写括号**,哪怕参数全默认也要 `@retry()`。

**延伸:类装饰器(Day 2 · 2026-04-19)**

**是什么**
用一个**类**当装饰器,`@Counter` 写在函数上面,装饰之后原来的函数名被替换成**类的实例**。调用这个"函数"其实是调用实例的 `__call__`。

```python
class Counter:
    def __init__(self, func):     # 装饰时调 1 次,保存原函数
        self.func = func
        self.count = 0
    def __call__(self, *args, **kwargs):   # 每次调用都走这里
        self.count += 1
        return self.func(*args, **kwargs)

@Counter
def say_hi(): print("hi")

say_hi(); say_hi(); say_hi()   # say_hi.count == 3
```

**为什么**
函数装饰器靠**闭包**记状态(`count` 藏在 wrapper 外层变量里),类装饰器直接用 `self.xxx` 存状态,**更直观、能带方法**(比如 `say_hi.reset()`)。适合"装饰器本身需要复杂状态 / 暴露操作接口"的场景。

**怎么用**
- `@Counter` **无括号**:`say_hi = Counter(say_hi)` —— `__init__` 直接接函数
- `@Counter(config)` **有括号带参**:`say_hi = Counter(config)(say_hi)` —— 这时候 `__init__` 接配置,需要把接函数的工作挪到 `__call__`,骨架变成:

```python
class Retry:
    def __init__(self, times):    self.times = times
    def __call__(self, func):
        def wrapper(*args, **kw): ...
        return wrapper
```

注意 `@Retry(times=3)` 和 `@Counter` 是两种不同的写法,前者 `__call__` 接函数,后者 `__init__` 接函数。

**坑在哪**
- **以为"每次调用都重新创建装饰器实例"** —— 错。`@Counter` 只在**模块加载时执行一次**,之后 `say_hi` 就是那个**固定实例**,count 会一路涨。
- `@Counter` 加括号变 `@Counter()` 会报 `TypeError: missing 1 required positional argument: 'func'` —— 因为 `Counter()` 是无参调用,但 `__init__` 要求 `func`。本质是 Day 2 同一个坑:**Python 不检查语义类型,只检查参数个数**。

**延伸:装饰器叠加顺序(Day 2 · 2026-04-19)**

**是什么**
多个 `@` 叠起来:

```python
@A
@B
def foo(): ...
```

等价于 `foo = A(B(foo))`。**离函数近的先装**(B 先装 foo,A 再装 "B 包过的 foo"),最终 `foo` 这个名字指向**A 的 wrapper**。

**为什么**(洋葱模型)
调用 `foo()` 时:

```
进入 A.wrapper → "A before"
  进入 B.wrapper → "B before"
    进入 原 foo  → "hello"
  退出 B.wrapper → "B after"
退出 A.wrapper → "A after"
```

输出:

```
A before
B before
hello
B after
A after
```

**装饰从内到外,调用从外到内进洋葱、从内到外出洋葱。**

**怎么用**
- `@timing` + `@retry` 一起装:
  ```python
  @timing      # 外层,统计含重试在内的总耗时
  @retry(3, 1)  # 内层,真正执行重试
  def fetch(): ...
  ```
  相当于 `fetch = timing(retry(3,1)(fetch))` —— 顺序会影响你统计的是"一次调用"还是"含重试的总耗时"。
- 记口诀:**想最先/最后看到的那层日志,写在最上面**。

**坑在哪**
- 以为"A 装一次 hello,B 装一次 hello,各跑各的" —— **错**。A 装的是 **B 装过的那个对象**,原 hello 只会被调**最里那一次**。
- 日志混乱排查时,先画 `f = A(B(C(f)))` 展开式,再从外到内走一遍。

---

**附:`try/except` 不是 `if/else`(Day 2 · 2026-04-19)**

这是 `@retry` 写重试循环时栽的跟头。对比:

| | if / else | try / except |
|---|---|---|
| 选择方式 | **主动判条件**,二选一 | **默认走 try**,抛异常才跳 except |
| 不抛异常 | — | except **一次都不执行** |
| try 里 return | — | 能。return 成功就**立即退出函数**,后面代码全不跑 |

所以 `try: return func(); except: ...` 在 retry 里的语义是:**成功就立刻退出,失败才重试**。不是"每次都走一遍 except"。

另:**`return` 一执行,函数立即结束**,`return` 后面那行叫 **unreachable code**(死代码),永远到不了。

**附:闭包修改外层变量的陷阱(Day 2 · 2026-04-19)**

```python
counter = 0
def flaky():
    counter += 1   # 💥 UnboundLocalError
```

**为什么炸**:Python 一看到函数体内对 `counter` 有**赋值**(`+= 1` 含隐式赋值),就把 `counter` 当**局部变量**处理。执行 `counter + 1` 时它还没值 → 炸。

**两种修法**:

```python
# 方法 1:用可变容器(dict/list),只改内容不赋名
counter = {"n": 0}
def flaky():
    counter["n"] += 1   # ✅ 没给 counter 重新赋值,只改字典内容

# 方法 2:nonlocal 显式声明"这个变量来自外层"
def outer():
    counter = 0
    def flaky():
        nonlocal counter
        counter += 1
```

这不是偏门语法,是 Python 闭包的基本规则 —— 以后写装饰器 / 生成器 / 协程会反复用到。

#### 生成器(Day 3 · 2026-04-19)

**是什么**
generator 就是**一个可暂停、可恢复、带状态的函数**。识别方法:**函数体里出现 `yield` 关键字**(哪怕一次),这个函数就自动变成 generator。调用它**不会执行函数体**,只返回一个 generator 对象——相当于一个"遥控器",每按一次 `next()`,它才醒过来跑到下一个 yield,把值吐出来,**原地冻结**。

```python
def gen():
    print("A")
    yield 1       # 吐 1,冻结
    print("B")
    yield 2       # 吐 2,冻结
    print("C")    # 最后一次 next 时才跑,然后抛 StopIteration
```

**为什么需要**
`return` 是一次性把整个结果装进内存再吐,遇到 10GB 大文件 / 无限序列 / LLM 流式输出(token 一个个来)直接破产。generator 的**惰性求值**——要一个给一个——让同一时刻内存里只有当前那一个元素。这是后面 SSE 流式响应、RAG 文档流、asyncio 协程的共同底座。

**怎么用(三大模式)**

1. **单独用作数据流**:
   ```python
   def read_lines(path):
       for line in open(path):
           yield line.rstrip("\n")   # 一行一行吐
   ```

2. **generator 串 generator**(管道):
   ```python
   def filter_contains(lines, keyword):
       for line in lines:
           if keyword in line:
               yield line
   pipeline = filter_contains(read_lines("access.log"), "ERROR")
   ```
   数据只是"流"过,全程不落地。

3. **`yield from` 透传**:
   ```python
   def outer():
       yield 0
       yield from inner()    # 把 inner 吐的每个值当自己吐的
       yield 99
   ```
   **注意选型**:想"整批透传"用 `yield from`;想"过滤 / 变换"用 `for + if + yield`。别乱用。

**坑在哪(自己踩过的 5 个)**
1. **函数里没 yield 就不是 generator** —— 只写 `print(line)` 忘了 `yield`,它就是个返回 `None` 的普通函数
2. **yield 是暂停键,不是终止键** —— `yield X` 后面的代码**要下一次 next 才跑**。所以 `yield line; line.rstrip()` 里 rstrip 永远作用在已经吐出去之后,对那次输出无效
3. **str 不可变** —— `line.rstrip("\n")` 不改 `line` 本身,返回新字符串。不接住返回值 = 白跑。一律写 `yield line.rstrip("\n")` 合成一步
4. **generator 只能遍历一次** —— 用完即废,想再来一遍得重新调 `gen()` 拿新对象
5. **`for x in 3` 会报错** —— int 不可迭代,要 `for x in range(3)`。Python 的 for 是"遍历可迭代对象",不是 C 风格的 `for(i=0;i<n;i++)`

**附:for 循环的脱壳**

```python
for x in g:
    do(x)
# 等价于
g = iter(g)
while True:
    try:
        x = next(g)
    except StopIteration:
        break
    do(x)
```

`for` = 自动 next + 自动捕获 StopIteration。

**附:数 generator 元素个数的通用模板**

```python
count = sum(1 for _ in some_generator)
```

原理:每遍历一次贡献一个 1,总和 = 元素个数。`_` 表示"这个变量我不关心"。之所以不能 `len(generator)`,因为 generator 自己也不知道有多少——得吐完才知道。

**一句话记忆**
**yield 是暂停键,`yield from` 是转发键,generator 是管道里的一节水管**。

#### 迭代器协议(Day 4 · 2026-04-19)

**是什么**
Python 里"能被 for 遍历"这件事靠**迭代器协议**支撑——一个两方合同:

```python
__iter__(self)    # 返回一个"迭代器对象"
__next__(self)    # 返回下一个元素;没了就 raise StopIteration
```

`for x in obj` 的底层展开:

```python
it = iter(obj)                 # 调 obj.__iter__()
while True:
    try:
        x = next(it)           # 调 it.__next__()
    except StopIteration:
        break
    do(x)
```

**为什么需要**
标准化"遍历"这件事,让 list / tuple / dict / str / 文件 / generator / 自定义 class...... 全部可以用同一个 `for` 语法。学会这一套还能看懂 LangChain / PyTorch / Django 源码里大量 class 形式的迭代器(数据集、批处理器、流式响应)。

**核心区分:Iterable vs Iterator**(今天最容易混的点)

| | Iterable(可迭代对象) | Iterator(迭代器) |
|---|---|---|
| 实现了什么 | 只 `__iter__` | `__iter__` + `__next__` |
| 能 `for` | ✅ | ✅ |
| 能 `next()` | ❌ | ✅ |
| 典型例子 | list / tuple / dict / str / range | generator / `iter(lst)` 返回的、文件对象、自己写的 class |
| 能多次遍历 | ✅(每次 for 都 `iter()` 新造一个游标) | ❌ 用完即废 |

记忆锚:
- **Iterable = 数据本体**(像一本书,能被翻很多遍)
- **Iterator = 一次性游标**(像书签,用完撕掉)
- `iter()` = 从书里抽一张新书签
- `next()` = 书签往后挪一页
- `StopIteration` = 翻到尽头

**判 Iterator 的铁律**:`iter(x) is x` 为 True ⟺ x 是 Iterator。所以自己写 Iterator 时,`def __iter__(self): return self` 是**固定套路**,不是随便写。

**怎么用(class 三段式)**

```python
class Fib:
    def __init__(self):
        self.a, self.b = 0, 1
    def __iter__(self):
        return self                                 # Iterator 必须返回自己
    def __next__(self):
        value = self.a
        self.a, self.b = self.b, self.a + self.b   # 更新状态
        return value
```

**generator 和迭代器协议的关系**
generator 对象**自动满足 Iterator 协议**——有 `__iter__` / `__next__`,`iter(g) is g` 为 True。所以 yield 其实就是迭代器协议的**语法糖**:Python 帮你自动做"__iter__ 返 self、状态冻结/恢复、跑到底抛 StopIteration",你只管写 yield。

**class vs generator 选型**
| 场景 | 选 |
|---|---|
| 简单吐一串值 | **generator**(行数 1/2 到 1/3,不用手写骨架) |
| 需要暴露 `.reset()` / `.progress()` 这种额外方法 | **class Iterator**(generator 对象状态封闭,改不了) |
| 封装"集合对象",想让别人能**多次遍历** | **class Iterable**(`__iter__` 返回新 Iterator,像 list 那样) |

**坑在哪(自己踩过的 3 个)**
1. **把 list 当 Iterator** — `next([1,2,3])` 会炸(list 没 `__next__`),要先 `iter([1,2,3])`
2. **class Iterator 的 `__next__` 里写 `while True`** — 错。`__next__` 每次只返回一个值,循环由外部 for 反复调 next 驱动
3. **generator 里不写 `while True`** — 错反了!generator 函数跑到函数体底就抛 StopIteration,要吐无穷序列必须 `while True: yield`。**class 不写循环,generator 必须写循环,位置正好相反**

**附:Iterator 用一次就废的真实影响**

```python
g = fib_gen()
for x in g: print(x)    # 正常吐
for x in g: print(x)    # 什么都不打,g 已经耗尽
```

想再遍历一遍:**重新调 `fib_gen()` 拿新对象**,或者把它封成 Iterable(`__iter__` 返回新 Iterator)。

**一句话记忆**
**迭代器协议 = `__iter__` + `__next__` + StopIteration**;**Iterable 是书,Iterator 是书签**;**generator 是迭代器的语法糖**——class 不写循环、generator 必须写循环,两者位置相反。

#### 上下文管理器(Day 5 · 2026-04-19)

**是什么**
`with` 语句是 **try/finally 的语法糖**,专门解决"资源获取后必须释放"的场景 —— 文件、数据库连接、socket、锁,只要"拿了必须还",都走 with。任何对象只要实现**上下文管理器协议**(`__enter__` + `__exit__`)就能塞进 with 里。

```python
# 这两段行为完全等价
with open("a.txt") as f:        # 语法糖
    data = f.read()

f = open("a.txt")               # 等价的手动写法
try:
    data = f.read()
finally:
    f.close()
```

**为什么需要**
不用 with 直接 `f = open(); f.read(); f.close()` 有两个坑:(1) 中间抛异常 → close 永远跑不到 → 文件句柄泄漏;(2) 函数里多个 return 分支,每个 return 前都得手动 close,极易漏。with 把"保证清理"这件事从程序员责任变成**语言责任**。

**两种写法(核心对比)**

**A 版:class 形式**

```python
class MyResource:
    def __enter__(self):
        # 进入 with 时自动调;返回值就是 as xxx 的 xxx
        return self
    def __exit__(self, exc_type, exc_val, exc_tb):
        # 离开时自动调(正常/异常/return/break 都会触发)
        # exc_type is None 表示无异常
        ...
```

**B 版:`@contextmanager` 装饰器**

```python
from contextlib import contextmanager

@contextmanager
def my_resource():
    # yield 之前 = __enter__
    try:
        yield "hello"        # yield 的值就是 as xxx 的 xxx
    except Exception:
        # 异常分支
        raise                # ⚠️ 必须显式 raise,不然吞异常
    else:
        # 无异常分支
        pass
    finally:
        # yield 之后 = __exit__,总是跑
        pass
```

**两版机制对比(今天最关键的认知表)**

| 维度 | class 版 | `@contextmanager` 版 |
|---|---|---|
| `as xxx` 赋的值 | `__enter__` 的返回值 | yield 后面的表达式 |
| **怎么感知异常** | `__exit__` 的 `exc_type` 参数(None=无异常) | `try/except` 在 yield 这一行捕获 |
| **怎么让异常向上抛** | 默认 `return None` 就会抛 | `except` 必须显式 `raise` |
| 清理代码写哪 | `__exit__` 方法体 | `finally` 块 |

**两版行为等价,但机制路径完全不同**。记住这对应关系,后面读 SQLAlchemy session、httpx client、FastAPI lifespan 的源码就不会迷路。

**选型原则**
- 资源需要跨多个方法共享 / 想暴露 `.elapsed()` 之类接口 → **class**
- 简单的"进入-退出",状态内聚在一个函数里 → **`@contextmanager`**(省一半代码)
- 精细处理不同类型异常(`if exc_type is XXX`) → **class**(直接拿到 exc_type)

**坑在哪(自己踩过的 5 个)**

1. **dunder 方法拼错 Python 不报错** —— `__intit__` 不是 `__init__`,但 Python 对它来说只是个"名字普通的方法",默默忽略。实例化时用的是默认 `object.__init__`,`self.name` 根本没被设置 → 进 `__enter__` 就 `AttributeError`。以后写 `__enter__` / `__exit__` / `__call__` / `__iter__` 每次都**盯一眼拼写**。
2. **`__exit__` return True 会吞异常** —— 默认 return None 正好让异常向上抛,这是我们要的。别随便 return True,除非明确要压制某类异常。
3. **`@contextmanager` 里 yield 必须包 try/finally** —— 不包的话,with 块抛异常 → 异常在 yield 这一行重抛 → 后面所有清理代码(commit/rollback/close)全跑不到 → **清理漏做**。
4. **except 忘了 raise → 异常被静默吞掉** —— with 上层的 `except RuntimeError: caught: oops` 根本抓不到,看起来"一切正常"实际业务已经炸了。**必须显式 raise**。
5. **裸 `except:` 会吞 Ctrl-C** —— 和 Day 2 `@retry` 作业的坑同源。默认用 `except Exception:`,想捕获所有(含 KeyboardInterrupt / SystemExit)才用 `except BaseException:`。

**附:f-string 小坑(作业顺手摔的)**
`{}` 里**只放变量/表达式**,其他都是字面字符。想打印 `[DB] exec: INSERT` 应该写 `f"[DB] exec: {sql}"`,**不要** `f"{exec: <sql>}"`(后者会把 exec 当内置函数、`: <sql>` 当格式说明符,直接炸)。

**一句话记忆**
**`with` = 自动 try/finally**;class 版 `__exit__` 默认 return None 透传异常,`@contextmanager` 版 `except` 必须显式 `raise` 透传,两版行为一样但机制不同。

#### 魔法方法 + 动态属性(Day 6 · 2026-04-20)

**是什么**
Python 的"魔法方法"(dunder methods)是一组约定好**名字带双下划线**的方法,Python 解释器在执行特定语法时会自动调用它们。今天聚焦 4 个:

| 方法 | 触发时机 | 作用 |
|---|---|---|
| `__init__(self, ...)` | `Foo(...)` 实例化 | 初始化实例属性 |
| `__call__(self, ...)` | `obj(...)` 把实例当函数调 | 让实例变成"可调用物"(装饰器类、模型、策略) |
| `__repr__(self)` | `print(obj)` / `f"{obj}"` / list 里显示 / REPL 回显 | 对象的调试字符串 |
| `__eq__(self, other)` | `obj == xxx` | 自定义相等规则 |
| `__getattr__(self, key)` | `obj.xxx` **正常查找失败**的兜底 | 动态属性、代理、点取式 dict |

**为什么需要**
1. 不重写 `__repr__`:`print(obj)` → `<__main__.Foo object at 0x000...>`,看不出里面装了啥
2. 不重写 `__eq__`:两个"长得一样"的对象 `==` 结果是 `False`(默认等价于 `is`,比地址)
3. 不重写 `__getattr__`:没法搞 `cfg.database.host` 这种点取式 —— 只能 `cfg["database"]["host"]`,括号引号一堆

**这 4 个方法让自定义类能"融入 Python 原生语法"**。`print`、`==`、`()`、`.` 这些操作符对你的类有意义,类就真正变成"一等公民"。

**属性查找的 5 步**(理解 `__getattr__` 的前提)
```
obj.xxx 触发:
1. 找实例 obj 自己的 __dict__ → 有就返回
2. 找类 type(obj) 的 __dict__      → 有就返回
3. 顺着 MRO 找父类                 → 有就返回
4. 都没找到 → 调 __getattr__(self, 'xxx')
5. __getattr__ 也没定义 / 内部 raise AttributeError → 最终抛 AttributeError
```

**关键点**:`__getattr__` 是"**查找失败**的兜底",**不是每次访问都触发**。这跟同名兄弟 `__getattribute__`(每次访问都触发,第一入口)是完全不同的东西。日常只写 `__getattr__`,因为 `__getattribute__` 里随便访问 `self.x` 就会再次触发自己 → 死循环。

**怎么用(Config 类核心模式)**

```python
class Config:
    def __init__(self, data):
        # 递归包装嵌套 dict,让 .a.b.c 能层层点下去
        self._data = {k: (Config(v) if isinstance(v, dict) else v) for k, v in data.items()}

    def __getattr__(self, key):
        try:
            return self._data[key]
        except KeyError:
            raise AttributeError(f"{type(self).__name__!r} object has no attribute {key!r}") from None

    def __repr__(self):
        parts = ", ".join(f"{k}={v!r}" for k, v in self._data.items())
        return f"Config({parts})"    # {v!r} 让嵌套 Config 自动走自己的 __repr__ 递归展开

    def __eq__(self, other):
        if not isinstance(other, Config):
            return NotImplemented     # ⚠️ 不是 return False
        return self._data == other._data
```

**核心套路记忆**
- `__init__` 里递归包装嵌套 dict(不要留到 `__getattr__` 里现包,会有身份不稳定 / repr 爆炸的问题)
- `__getattr__` 用 **EAFP**(try/except)比 **LBYL**(if in + [])更 Pythonic 且命中路径只查 1 次
- `__repr__` 里用 `{v!r}` 调 repr,嵌套对象自动递归展开;字符串自动带引号
- `__eq__` 类型不匹配 **`return NotImplemented`**(大写常量,不是异常),不要 `return False` —— 会破坏对称性
- `AttributeError` 消息用 Python 内置风格:`f"{type(self).__name__!r} object has no attribute {key!r}"`

**Config 类的现实对应物**:Pydantic `BaseModel`、OmegaConf `DictConfig`、HuggingFace `AutoConfig`、`argparse.Namespace` —— 底层套路都是这个。这道作业是写迷你版。

**坑在哪(自己踩过的 5 个 + /simplify 抓到的 1 个)**

1. **`if v:` 判真假替代"存在性检查"** —— `.get(key)` 拿到 0 / False / "" / [] 都会被错判为"不存在",然后误抛 AttributeError。应该用 `if key in self._data` 或更地道的 try/except。**核心:判断"键存在"和"值为真"是两件事**。
2. **`.values` 漏括号** —— `self._data.values` 是方法对象本身,不是值列表。两个 dict 的 `.values` 方法是两个不同对象,永远不等。比较 dict 直接 `self._data == other._data` 就行,Python 内置逐项比较。
3. **`return NotImplemented` ≠ `return False`** —— `NotImplemented` 是一个**大写常量**(不是 NotImplementedError 异常!)。告诉 Python:"我不会跟这玩意比,让右边对象试试"。直接 return False 的话,`other == self` 会和 `self == other` 结果不一致,破坏对称性。
4. **`__getattr__` 无限递归的经典坑** —— 方法里访问 `self._data`,如果 `_data` 没被 `__init__` 设置(pickle / copy / `__new__` 绕过 init 的边界场景),访问 `self._data` → 走属性查找流程 → 找不到 → 调 `__getattr__` → 又访问 `self._data` → ... RecursionError。生产代码防御:`if key == "_data": raise AttributeError(key)` 在方法第一行守护。
5. **`AttributeError(key)` 消息太短** —— Python 内置的 AttributeError 消息是 `'Config' object has no attribute 'xxx'`,想跟它对齐要手动拼 `f"{type(self).__name__!r} object has no attribute {key!r}"`。不拼也能用,但调试时看起来光秃秃。
6. **LBYL vs EAFP**(/simplify 抓的)—— Python 哲学倾向 **EAFP**(Easier to Ask Forgiveness than Permission)。`if key in d: return d[key]` 命中路径要 2 次 dict 查询,`try: return d[key] except KeyError` 命中只要 1 次。而且 EAFP 没有 TOCTOU(时序竞争)问题。配合 `raise ... from None` 清理异常链,traceback 更干净。

**附:途中顺手学的 3 个 Python 概念**

- **闭包 + nonlocal**:内层函数引用外层函数变量 = 闭包;内层**修改**外层变量必须 `nonlocal`,否则 Python 按默认规则创建局部变量 → `UnboundLocalError`。Day 2 `@retry` 用字典 `{"n": 0}` 绕过就是因为字典是"改内部",不是"重新赋值"。
- **字典推导式 + 三元表达式**:`{k: (A if cond else B) for k, v in d.items()}` —— 一行完成"遍历 + 条件改造"。Python 三元表达式顺序跟 C 系反过来:**先结果、再 if 条件、再 else 备选**。
- **f-string `!r` vs 默认**:`{x}` 调 `str(x)`,`{x!r}` 调 `repr(x)`。字符串在 `!r` 下会**带上引号**,数字不变。调试日志、报错消息优先用 `!r` 让类型一目了然("abc" 和 abc 变量名能立刻区分)。

**一句话记忆**
**魔法方法 = Python 语法糖的接插件**;`__getattr__` 是"查找失败后的兜底",**不是每次访问都触发**;EAFP 优于 LBYL;`__eq__` 类型不匹配用 `return NotImplemented`,重写 `__eq__` 会自动把 `__hash__` 设成 None(对象不能进 set / dict key)。

#### Week 1 综合复盘(Day 7 · 2026-04-21)

**是什么**
Week 1 6 天 6 个独立主题(装饰器 / 生成器 / 迭代器 / 上下文管理器 / 魔法方法 / 类型注解)在 Day 7 合到一起:10 道面试题检验概念,再写一个 `tinylog` 小工具——**同时用上装饰器 + 上下文管理器 + 生成器**三种机制,对应 Week 1 学过的骨架。这不是新知识点,是把碎片拼成能跑的东西。

**为什么**
光会"讲清楚"没用,面试要求"现场写"。6 天单点练习解决的是"这个 API 怎么用",综合练习解决的是"三个机制各管什么、边界在哪、组合起来哪儿会冲突"。结果一写就露馅:装饰器第一版结构完全错了,两次重写才对齐。

**怎么用**

`tinylog.py` 三件套骨架:

```python
import functools, time
from contextlib import contextmanager
from pathlib import Path
from typing import Iterator

def log_calls(func):
    @functools.wraps(func)
    def wrapper(*args, **kwargs):
        parts = [repr(a) for a in args] + [f"{k}={v!r}" for k, v in kwargs.items()]
        print(f">>> {func.__name__}({', '.join(parts)}) called")
        start = time.perf_counter()                   # ⚠ start 放这,不是放 log_calls 里
        try:
            result = func(*args, **kwargs)
        except Exception as e:
            elapsed = time.perf_counter() - start
            print(f"!!! {func.__name__} raised {type(e).__name__}: {e} after {elapsed:.4f}s")
            raise                                     # ⚠ 重抛,不吞
        elapsed = time.perf_counter() - start
        print(f"<<< {func.__name__} returned {result!r} in {elapsed:.4f}s")
        return result                                 # ⚠ 必须把原返回值交还
    return wrapper

@contextmanager
def log_block(label):
    print(f"[{label}] start")
    start = time.perf_counter()
    try:
        yield
    except Exception as e:
        elapsed = time.perf_counter() - start
        print(f"[{label}] failed: {type(e).__name__}: {e} after {elapsed:.4f}s")
        raise                                         # ⚠ 同样要重抛
    else:
        elapsed = time.perf_counter() - start
        print(f"[{label}] end in {elapsed:.4f}s")

def tail(path: str | Path) -> Iterator[str]:
    with open(path) as f:
        for line in f:
            line = line.rstrip()
            if not line:
                continue
            yield line
```

组合驱动:`with log_block(...)` 套住 `for line in tail(...)` 循环,循环里 `process_line` 被 `@log_calls` 装饰 —— 三个机制各司其职,**Week 1 全体知识点一把激活**。

**坑在哪**

1. **装饰器 start 放错位置**(第一次最狠的坑):`start = time.perf_counter()` 放在 `log_calls(func)` 层,只在装饰那一刻执行一次,所以每次调用算出来的 elapsed 都是"自程序启动"不是"这次调用"。**start 必须放 wrapper 里**,和每次调用同生共死。
2. **`finally: return` 会吞异常**:异常正在向外传,`finally` 里 `return` 的瞬间 Python 会"吃掉"这个异常 —— 需求里明明要求"异常日志打完,异常要继续向外抛"全被这个 return 干掉。正确套路:异常路径 `except: ...; raise`,正常路径 `print; return result`,不用 finally 也能写。
3. **`func(*args, **kwargs)` 在 finally / 循环里重复调用**:原函数被执行两次。如果是 `send_email()` / `transfer_money()` 这种有副作用的函数 → 生产事故。任何装饰器都要**把 func 只调一次并 `result = ...` 存起来**。
4. **`except Exception: raise` 是无用模式**:抓了又立刻重抛 = 和没抓一样。写这个不是错,是"以为抓了能做点什么但实际什么也没做"。要么不抓,要么抓了打日志 + 重抛,要么抓了吞掉处理。
5. **`str` 不要做变量名**:shadow 内建类型。变量名避开 `str` / `list` / `dict` / `type` / `id` / `input`。
6. **`__name__` 不是函数**:`func.__name__(*args, **kwargs)` 把字符串当函数调 → `TypeError`。`__name__` 是属性,类型 `str`。
7. **`__name__` / `__doc__` 是属性不是魔法方法**:属性存储函数名字符串 / docstring 字符串,`@functools.wraps(func)` 的唯一作用就是把这些属性从原 func 复制到 wrapper 上,不加就会被覆盖成 `wrapper` / `None`。
8. **`yield from` 能接收子生成器的 `return` 值**:`result = yield from inner()`,inner 的 `return "done"` 被塞进 `StopIteration.value`,被委托它的 `yield from` 捕获成 `result`。**for 循环看不到 return 的值**,只有 `yield from` 能拿到 —— 这是协程双向通信机制的前身。
9. **重写 `__eq__` 后 `__hash__` 自动变 None**:Python 不变量 "`a == b → hash(a) == hash(b)`" 不能被破坏。你一改 `__eq__`,Python 就把 `__hash__` 设成 None 防你写出不一致代码。后果:对象不能进 `set` / `dict key`,`{u1, u2}` 直接 `TypeError: unhashable type`。要支持集合就必须**自己也重写 `__hash__`**,用**和 `__eq__` 同一组字段**:

   ```python
   def __eq__(self, other):
       return isinstance(other, User) and self.name == other.name
   def __hash__(self):
       return hash(self.name)   # 和 __eq__ 用的字段对齐
   ```

10. **生成器 + `with open(...)` 的文件生命周期陷阱**:`tail` 把 `with open(path) as f` 放进生成器里 —— 文件什么时候关?**不是 caller `for` 结束就关**,而是生成器**耗尽** / **GC 回收** / **显式 `.close()`** 才关。极端情况:`g = tail(p); next(g)` 拿一行就丢,文件会一直占着直到 GC。规避写法:把 `with open` 交给 caller,生成器只接一个 file-like 对象。

**`/simplify` 7 条全应用**

- 缩进 3→4 空格(PEP 8)
- 异常日志 `raised {e}` → `raised {type(e).__name__}: {e}`(带类型,调试不抓瞎)
- `tail` 里 `lines` → `f`(file handle 惯例)
- `tail` 加类型注解 `def tail(path: str | Path) -> Iterator[str]:`
- `log_block` 用 `try / except / else` 替代 `try / except`,`elapsed` 每路径只算一次(可读性 + 减少重复)
- demo `"test.log"` → `Path(__file__).parent / "test.log"`(脚本位置无关,之前被 cwd 坑过)
- args 格式化从 `add(args = (1, 2), kwargs = {}) called` → `add(1, 2) called`(自然调用样式,`repr(a) for a in args` + `f"{k}={v!r}"`)

**类型注解补课(Q11 答不上来的那个)**

Python 是**动态类型**,类型注解**运行时不做任何检查** —— `def greet(name: str)` 传 `123 + 456` 照样正常返回 `580`。注解存在的意义是:(1) IDE 补全 / 警告;(2) mypy / pyright 静态检查器在 CI 里拦 bug;(3) 文档作用;(4) Pydantic / FastAPI / dataclasses 这些框架**运行时主动读**注解做事(校验、路由解析、字段初始化)—— 但那是库的行为,不是 Python 的行为。

`list[int]` vs `List[int]`:Python 3.9+ 直接用内建 `list[int]` / `dict[str, int]` / `tuple[int, ...]`,不用从 `typing` import;3.8 及更早必须 `from typing import List` 用 `List[int]`。现在优先用小写内建。`Optional` / `Union` / `Callable` 还是从 `typing` 来;3.10+ `int | str` 替代 `Union[int, str]`,`str | None` 替代 `Optional[str]`。

**一句话记忆**
Week 1 = **装饰器是"函数包装" + 生成器是"暂停恢复的函数" + 上下文管理器是"带自动清理的代码块" + 魔法方法是"Python 语法糖的接插件"**;组合时的铁律是**"异常该抛就抛,`finally` 里不要 `return`,装饰器只调 func 一次并记得 `return result`"**。

### Week 2 · 异步 + 工程化

#### asyncio 基础(Day 8 · 2026-04-21)

**是什么**

Python 的**单线程并发**方案:一个线程配一个事件循环(event loop),协程挂起等 I/O 的时候把 CPU 交给别的协程 —— 看起来像"同时做",这叫**协作式调度**(cooperative scheduling)。语法四件套:

| 关键字 / API | 干什么 |
|---|---|
| `async def foo()` | 定义协程函数。**调用 `foo()` 不会执行**,只返回一个协程对象,必须被 `await` / `gather` / `create_task` 才真正跑 |
| `await x` | 等 `x` 完成,期间让出 CPU 给别的协程。只能在 `async def` 里用 |
| `asyncio.run(协程)` | 启动事件循环,跑这个协程到结束。整个程序通常只调一次 |
| `asyncio.gather(*coros)` | 并发跑多个协程,等全部完成,**按传入顺序**返回结果列表 |

**为什么需要**

CPU 比网络 / 磁盘快几个数量级。同步代码里 CPU 大部分时间在睡觉等外部响应 —— 爬 10 个 1s 的网页,CPU 忙 0.1s 闲 9.9s。异步把 10 段"等"**重叠**起来:**墙上时间从 O(N) 压到 O(1),但 CPU 工作量没变**。

所以判断"该不该用异步"看的是任务类型:

- **I/O 密集**(爬虫、LLM API、DB 查询、Redis、文件读写)→ ✅ 异步能加速
- **CPU 密集**(矩阵运算、加密、压缩、纯 pandas 计算)→ ❌ 异步救不了,要上多进程

一句话记忆:**异步治"等待"不治"算得慢"**。

**怎么用(Day 8 作业的终版代码)**

```python
import asyncio
import httpx
import time

async def fetch(client, url):
    r = await client.get(url)
    return r.status_code

async def main():
    urls = ["https://www.example.com"] * 10
    async with httpx.AsyncClient() as client:   # 共享一个 client 复用连接池
        start = time.perf_counter()
        results = await asyncio.gather(*(fetch(client, url) for url in urls))
        print(results)
        print(f"耗时 {time.perf_counter() - start:.2f}s")

asyncio.run(main())
# 输出 [200]*10,耗时 1.25s(同步版 ≈ 20s)
```

**坑在哪**

1. **`async def foo()` 调用不自动执行 —— 头号坑**。调用只返回一个协程对象,不 `await` 不 `gather` 不 `create_task` 就永远不跑,Python 会警告 `coroutine 'foo' was never awaited`。这是"`async def` 和普通 `def` 的最大行为差"。我在 Q4 的 C 版本和后面作业各踩一次才记住。

2. **顺序 `await` 多个协程 = 串行,和同步代码一模一样**。光写 `async` / `await` 不加速,真正让并发发生的是 `gather` / `create_task` 这种**并发调度手段**。
   ```python
   await task1()   # 等 task1 跑完
   await task2()   # 再跑 task2 —— 完全串行
   ```

3. **`gather(iterable)` 和 `gather(*iterable)` 天差地别**。`gather` 的签名是 `*coros_or_futures`,它要的是**多个位置参数**,不是一个容器。
   ```python
   asyncio.gather(coros)       # ❌ 只传 1 个参数(一个 list)
   asyncio.gather(*coros)      # ✅ 拆成 N 个位置参数
   ```

4. **异步 I/O 库不能混同步调用**。`async def fetch` 里写 `httpx.get(url)`(同步)会**阻塞整个事件循环**,10 个请求退化成串行。异步世界里所有 I/O 必须是异步版本(`await client.get(url)`)。

5. **`httpx.AsyncClient` 是类,要实例化**。写 `async with httpx.AsyncClient as client` 会崩 —— 类本身不是异步上下文管理器,必须 `AsyncClient()` 调用一次。对照 `with open("x") as f:` 记忆,`open()` 也是调用不是类名。

6. **共享 client 是异步客户端的工程默认**。每个请求开新 client → 每次重建 TCP + TLS 握手,10 个请求 = 10 套连接池,亏爆。正确姿势:外层 `async with client:` 创建,把 client 透传给各 worker。这是 httpx / aiohttp / redis.asyncio / aiomysql 的通用模式,OnCall Agent 里会反复用。耗时从 2.13s → 1.25s,提速 41%。

**顺带搞懂的 `*` / `**` 解包**

两种星号,四个用法:

```python
# ① 调用时解包 iterable → 位置参数
f(*[1, 2, 3])         # 等价于 f(1, 2, 3)

# ② 定义时收集多余位置参数 → tuple
def g(*args): ...     # g(1,2,3) 则 args = (1, 2, 3)

# ③ 赋值时的"剩余解包"
first, *rest = [1, 2, 3, 4]    # first=1, rest=[2,3,4]

# ④ list / dict 字面量合并
c = [*a, *b, 5]
```

`**` 是字典 / kwargs 版本:`client.get(url, **kwargs)`。

**关键对照表**(这个分不清就全乱套):

| 写法 | 含义 | args 是什么 |
|---|---|---|
| `g(1, 2, 3)` | 传 3 个位置参数 | `(1, 2, 3)` |
| `g(*[1, 2, 3])` | 解包 list,等价于传 3 个 | `(1, 2, 3)` |
| `g([1, 2, 3])` | 传 1 个参数,这个参数是 list | `([1, 2, 3],)` ← 坑 |

**调用处的 `*` 和定义处的 `*` 互为逆操作**:调用处是"拆",定义处是"合"。Q6(3) 就栽在混淆这点上 —— `g([1,2,3])` 的 args 是 `([1,2,3],)`,`sum` 对它做 `0 + [1,2,3]` 直接 `TypeError`。



#### async 深入:Task / Semaphore / wait_for / cancel / Queue(Day 9 · 2026-04-21)

**是什么**

Day 8 学了"并发"的发动机(`gather`),Day 9 学的是"**控制**"这台发动机的五个旋钮:

| 旋钮 | 作用 |
|---|---|
| `create_task` | 起步发车(立刻调度),你想等就 `await task`,不想等就让它自己跑(fire-and-forget) |
| `Semaphore(N)` | 同时最多 N 个令牌,控制**并发数** |
| `wait_for(coro, timeout)` | 超时自动 cancel,抛 `asyncio.TimeoutError` |
| `task.cancel()` | 给 task 注入 `CancelledError`,触发清理 |
| `Queue(maxsize)` | 生产-消费解耦,背压保护 |

**为什么**

`gather` 之后的每一步都在解决"真实生产里会炸的问题":

- **并发太多炸对方**(429 / 封 IP / API 成本起飞)→ `Semaphore`
- **下游卡死拖垮自己**(线程池满 → 503)→ `wait_for`
- **用户要求中断 / 主程序要关后台 task** → `cancel`
- **生产消费速度不匹配**(LLM 慢、日志快)→ `Queue`

**怎么用**

**① Semaphore —— 甜点公式**

```
推荐并发数 ≈ rate × 平均耗时 × 0.5
```

打"五折"是为了留重试 buffer。四个典型场景:

| 场景 | 推荐并发 | 理由 |
|---|---|---|
| OpenAI 500 rpm(8.3 rps × 3s 响应) | 10 - 20 | rate × 耗时 = 25,打五折 |
| Prometheus 100 qps | 10 - 20 | 同上 |
| 本地 1000 JSON 文件(aiofiles) | 50 - 100 | FD 上限 + 线程池瓶颈 |
| MySQL 连接池 20 | 15 | 留 buffer 给其他调用方 |

模式:和 `AsyncClient` 一样是"外层创建,透传给 worker":

```python
sem = asyncio.Semaphore(3)
async def fetch(client, sem, url):
    async with sem:                  # 等令牌 → 干活 → 自动还
        r = await client.get(url)
        return r.status_code
```

**② wait_for —— 异常类型别抓错(坑)**

两种超时方式要抓不同的异常:

```python
# 库层超时(httpx)→ 抓 httpx.TimeoutException
async with httpx.AsyncClient(timeout=3.0) as client:
    try:
        r = await client.get(url)
    except httpx.TimeoutException:   # ← 不是 asyncio.TimeoutError
        return -1

# asyncio 层超时(wait_for)→ 抓 asyncio.TimeoutError
try:
    r = await asyncio.wait_for(client.get(url), timeout=3.0)
except asyncio.TimeoutError:
    return -1
```

**一句话**:谁负责超时,抓谁的异常。抓错类型 = 没抓,异常穿过 `gather` 整个崩掉。

**③ cancel —— 三条铁律**

```python
# 铁律 1:cancel 后必须 await task 让清理生效
task.cancel()
try:
    await task
except asyncio.CancelledError:
    pass

# 铁律 2:抓了 CancelledError 必须 raise
async def worker():
    try:
        await do_work()
    except asyncio.CancelledError:
        cleanup()
        raise          # ⚠️ 不 raise = 取消失效 = task 假装被取消其实继续跑

# 铁律 3:不要用 except Exception 抓异步代码(防误吞)
# Python 3.8+ 把 CancelledError 改成继承 BaseException 就是为了防这个
```

**wait_for 本质**:`create_task + sleep(timeout) + cancel + 抛 TimeoutError`,理解了 cancel 就懂了 wait_for。

**④ Queue —— 生产消费解耦 + 背压**

5 个核心 API:

| API | 作用 |
|---|---|
| `Queue(maxsize=N)` | 容量 N,满了 `put` 阻塞(**背压**) |
| `await queue.put(x)` | 塞入 |
| `await queue.get()` | 取出 |
| `queue.task_done()` | **"`done`"不是"`down`"**,消费者完成信号 |
| `await queue.join()` | 等所有 put 的都 task_done |

消费者是死循环,必须**外部 cancel** 或**毒丸信号**(put `None` 当结束)才能停。

**坑在哪**

1. **第三次漏 `await`**:`r = asyncio.wait_for(...)` 忘了 `await`,r 变协程对象,下一行调 `r.status_code` 崩。肌肉记忆:**看到 `asyncio.*` 就条件反射写 `await`**。

2. **`time.sleep` vs `asyncio.sleep`**:异步里 `time.sleep` 会**同步阻塞整个事件循环**,而且 `time.sleep` 返回 `None`,`await None` → `TypeError`。`time` 模块只留给 `perf_counter()` 计时,**所有 sleep 都用 `asyncio.sleep`**。

3. **API 名拼错两次同一个地方**:`task_done` 被先后拼成 `taskdown` 和 `task_down`。对,是"done"(完成)不是"down"(下去)。

4. **后台 task 异常静默丢失(最毒的坑)**:
   ```python
   cons = asyncio.create_task(consumer(...))
   await queue.join()    # 如果 consumer 里崩了,main 完全不知道,queue.join() 死等 task_done
   ```
   consumer 里一个 `AttributeError` 导致所有消费者静默死亡,生产者继续往满队列里 put(阻塞),`queue.join()` 等永远不会来的 `task_done` → **程序打了几行后无报错死锁**。工程防护:
   ```python
   async def safe_consumer(name, queue):
       try:
           await consumer(name, queue)
       except asyncio.CancelledError:
           raise
       except Exception as e:
           print(f"{name} 崩了:{type(e).__name__}: {e}")
           raise
   ```

5. **`await t1` 期间不是"只有 t1 在跑"**:所有已 `create_task` 的都在事件循环里跑,**完成顺序只看各自耗时**,跟你 await 谁无关(Q1 踩坑)。

**效果数据**

- 并发限流爬虫 30 URL(sem=3, timeout=3, 1 个故意超时):6s,29×200 + 1×(-1)
- 告警流 Queue(10 alert, 3 consumer, maxsize=2):round-robin 完美,消费者节奏主导整体(~3.3s)

**附带搞懂的工程词汇**

- **下游 / 上游**:数据流向 "上→下"。谁调我 = 上游(用户/前端),我调谁 = 下游(LLM / Loki / MySQL)。90% 故障话术围着"下游":"下游超时 / 下游限流 / 下游抖动"
- **线程池**:预创建的 worker 线程,任务来了挑空闲的跑。避免反复创建销毁。Python asyncio 默认线程池 = `min(32, CPU 核数 + 4)`。`aiofiles` 底层就是线程池模拟异步文件 I/O
- **HTTP 503**:Service Unavailable,服务端临时不可用(过载 / 维护 / 下游挂)。4xx 客户端锅、5xx 服务端锅;**429(限流)和 503(过载)是 OnCall 最常见两个告警**
- **解耦(decoupling)**:让模块不直接依赖彼此,改 A 不被迫改 B。Queue 是最典型的解耦工具 —— 生产者不知道谁消费,消费者不知道谁生产,中间靠队列接线

**消费速度决定系统吞吐**

Q6 作业印证的真理:**生产 0.2s/个、消费 1s/个、3 消费者** → 系统稳态吞吐 = 3 / 1 = 3 req/s,和生产速度无关。**加生产者不提速,加消费者才提速**。OnCall Agent 里 LLM 是最慢消费者 → Agent 总吞吐由 LLM 决定。
