- **状态**:Day 18 SSE 流式响应 ⭐ 完整收官 ✅。在 Day 17 中间件三件套基础上独立写出 60 行 SSE 端点:`POST /chat/stream` 用 `asyncio.Queue` 三角色协作(producer 反转 prompt 逐字推 + heartbeat 每 3 秒推 ping + consumer 主循环 yield SSE),客户端断开 `except CancelledError: log + raise / finally: cancel + gather` 闭环。**全天最大教学收获是 /simplify 抓的真 bug**:`asyncio.create_task` 启动的后台 task 异常**默认静默** —— producer 抛错被 task 对象 hold 住、不向外抛,consumer 卡 `await queue.get()` 永远 hang,只有心跳还在打,服务"看着活实则死"。修法是 producer 包 `try/except Exception` 把异常包成 `("error", exc)` 推 queue,consumer 加 error 分支显式收;`except Exception` 不抓 `CancelledError`(Python 3.8+ 继承 `BaseException`),正常 cancel 流程不受影响。**配套规律必须刻脑**:所有 `create_task` 都要考虑异常去哪儿 —— 这是 async 反直觉点(同步代码异常自动冒到上层 stack,async task 的异常被孤立在 task 自己肚子里)。次级认知:SSE 心跳必用 `: ping\n\n` 注释行(前端不触发 onmessage,只防中间设备超时);心跳间隔 ≤ min(各级超时)/2;done 必须 `break` 否则主循环卡死;事件块必须 `\n\n` 收尾(Q5 答对了写代码时还是踩了同款坑);`async def` 写出来 = 协程工厂,嵌套定义不会自动启动(Q12 揭示)。**手测全过**:短 prompt(2 字)无 ping、长 prompt(25 字)中间出现 2 次 `: ping`、Ctrl+C 服务端 loguru `WARNING client disconnected, prompt='...'`(`!r` 加引号识别)、回归测试 4 处简化改动后行为一致。/simplify 改 4 处全应用(HEARTBEAT_INTERVAL 提模块级 / done 改 b"" 字面量 / inline `reversed(prompt)` / **producer 加 try/except + consumer 加 error 分支** 关键修复)。新文件夹 `stage2_backend/day18_sse_streaming/`。
